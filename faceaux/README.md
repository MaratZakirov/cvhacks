# Head Pose Estimation from 5 Face Keypoints

Небольшой набор скриптов для оценки ориентации головы по 5 ключевым точкам лица: `nose`, `left_eye`, `right_eye`, `left_ear`, `right_ear`.
Проект состоит из двух файлов:

- `face2head.py` — офлайн-скрипт для отладки геометрии, визуализации 3D/2D точек и проверки решения `solvePnP` на одном примере.
- `face2head_live.py` — realtime-демо с вебкамерой и `YOLO pose`, которое оценивает положение головы, достраивает вспомогательные точки и рисует 3D-бокс под головной убор.

Скрипты полезны как стартовая база для задач:

- оценки head pose по редкому набору ключевых точек;
- геометрического достроения головы из 5 facial keypoints;
- визуализации направления взгляда/лица относительно камеры;
- примерки 3D-примитивов на голову, например зоны каски, шлема или другого PPE.

## Что делает каждый файл

### `face2head.py`

Этот файл нужен в первую очередь для **геометрической отладки**.
Он:

1. задаёт каноническую 3D-модель головы на 5 опорных точках;
2. задаёт дополнительные 3D-точки головы (`макушка`, `подбородок`, `лоб`, `основание шеи`, `затылок`);
3. задаёт 3D-вершины бокса для области головного убора;
4. визуализирует все эти точки в 3D через `matplotlib`;
5. принимает заранее заданные 2D-координаты ключевых точек на изображении;
6. решает задачу `PnP` через `cv2.solvePnP(..., flags=cv2.SOLVEPNP_EPNP)`;
7. проецирует дополнительные 3D-точки обратно в 2D;
8. вычисляет вектор направления лица `back_head -> nose`;
9. считает угол между этим вектором и оптической осью камеры.

Этот скрипт удобен, когда нужно вручную:

- крутить каноническую 3D-модель;
- смотреть, насколько правдоподобно расположены вспомогательные точки;
- подгонять размеры коробки под каску/шлем;
- проверять устойчивость решения `solvePnP` без realtime-части.

### `face2head_live.py`

Этот файл — уже **боевой интерактивный прототип**.
Он:

1. загружает модель `Ultralytics YOLO pose`;
2. читает кадры с вебкамеры;
3. выбирает лучшего человека в кадре по среднему confidence 5 head keypoints;
4. строит позу головы по этим 5 точкам;
5. проецирует вспомогательные точки головы в изображение;
6. проецирует 3D-бокс для головного убора;
7. рисует на кадре исходные keypoints, достроенные точки, рёбра бокса и служебную информацию;
8. показывает угол лица к нормали камеры и 3D-координаты некоторых точек в системе камеры.

По сути это минимальный pipeline:

`webcam -> YOLO pose -> 5 keypoints головы -> solvePnP -> 3D head reconstruction -> overlay`

## Идея геометрии

В основе лежит каноническая 3D-модель из 5 точек:

- `nose`
- `left_eye`
- `right_eye`
- `left_ear`
- `right_ear`

Именно эти точки ожидаются и в 2D после pose-модели.
Дальше предполагается, что между канонической 3D-моделью и наблюдаемыми 2D-точками можно найти такое вращение и перенос, которые лучше всего объясняют проекцию головы в кадр.

После получения `rvec` и `tvec` можно:

- перевести любые дополнительные 3D-точки головы в систему камеры;
- проецировать их в 2D;
- достраивать геометрические примитивы вокруг головы;
- оценивать направление головы относительно камеры.

### Вспомогательные точки

`PTS_3D_AUX` содержит дополнительные смысловые точки:

- `top_head`
- `chin`
- `forehead`
- `neck_base`
- `back_head`

Они не участвуют в `solvePnP`, а используются только после оценки позы — для визуализации и вычисления направлений.

### Бокс головного убора

`PTS_3D_HAT` задаёт 8 точек параллелепипеда над головой.
После оценки head pose этот бокс проецируется в кадр и может использоваться как:

- грубая 3D-область для каски;
- зона проверки присутствия helmet/hat;
- вспомогательная геометрия для дальнейшего CV-пайплайна.

## Как считается направление лица

После решения `PnP` все точки переводятся в систему координат камеры.
Далее берётся вектор:

- от `back_head`
- к `nose`

Этот вектор интерпретируется как направление лица.
После нормализации считается угол между ним и вектором `[0, 0, 1]`, то есть направлением вперёд вдоль оптической оси камеры.

Интерпретация простая:

- маленький угол — лицо ориентировано ближе к камере;
- большой угол — голова сильнее отвернута.

Важно понимать, что это не полноценные yaw/pitch/roll в человекочитаемом виде, а более простой и часто практичный признак: **насколько лицо смотрит в сторону камеры**.

## Как запустить

### 1. Отладка на одном примере

```bash
python face2head.py
```

Что произойдёт:

- откроется 3D-график с канонической моделью и дополнительными точками;
- откроется 2D-график с исходными ключевыми точками;
- после `solvePnP` откроется ещё один 2D-график уже с достроенными точками и боксом;
- в консоль выведется результат решения и угол направления лица к камере.

Это хороший режим, если нужно сначала убедиться, что сама геометрия в целом адекватна.

### 2. Realtime-режим с вебкамерой

```bash
python face2head_live.py
```

После запуска откроется окно `OpenCV` с live-видео.
На изображении будут нарисованы:

- 5 исходных head keypoints;
- дополнительные точки головы;
- зелёный 3D-бокс под головной убор;
- линия `back_head -> nose`;
- текстовая панель с технической информацией.

### Управление

Во время работы доступны клавиши:

- `q` — выход;
- `ESC` — выход;
- `m` — переключение `mps <-> cpu`.

## Как выбирается человек в кадре

Если в кадре несколько людей, используется функция `pick_best_person()`.
Она:

- берёт только 5 head keypoints;
- считает средний confidence по ним;
- выбирает человека с максимальным средним score.

Это очень простая, но в практике часто достаточная эвристика для demo/prototype-режима.
Если нужно, логику легко заменить, например на:

- самого крупного человека;
- человека ближе к центру кадра;
- человека с максимальным bbox area;
- трекаемого человека по ID.

## Камера и интринсики

Сейчас камера моделируется приближённо:

```python
fx = fy = w
cx = w / 2
cy = h / 2
```

Это быстрый инженерный компромисс, который часто работает удивительно неплохо для прототипов.
Но важно помнить:

- без реальной калибровки абсолютные 3D-координаты могут плавать;
- углы и масштаб зависят от выбранной канонической модели головы;
- при широкоугольной оптике ошибка может заметно расти.

Если нужна более точная геометрия, стоит:

1. откалибровать камеру;
2. использовать реальные `K` и `distCoeffs`;
3. при необходимости переопределить канонические 3D-точки под свою анатомическую модель.

## Ограничения текущего подхода

Проект специально сделан простым, поэтому у него есть ожидаемые ограничения.

### 1. Только 5 точек

`solvePnP` решается по очень маленькому числу опорных точек.
Если одно ухо или глаз определены плохо, оценка позы может заметно шуметь.

### 2. Каноническая модель условная

Координаты `PTS_3D`, `PTS_3D_AUX` и `PTS_3D_HAT` заданы вручную.
Они не претендуют на анатомическую точность и работают как инженерная форма-прикидка.

### 3. Нет фильтрации по confidence

В текущем виде все 5 точек используются как есть.
Даже если одна из них имеет слабый confidence, она всё равно участвует в `PnP`.

### 4. Нет temporal smoothing

В live-версии нет сглаживания по времени.
Поэтому при шумных keypoints бокс и вспомогательные точки могут дрожать.

### 5. Угол — это не yaw/pitch/roll

Считаемый `face_angle_deg` — это лишь угол между направлением лица и осью камеры.
Для UI/аналитики этого часто достаточно, но это не замена полноценным Euler angles.

## Что можно улучшить

Ниже список самых очевидных улучшений, если проект пойдёт дальше демо.

### Геометрия

- заменить `EPNP` на `solvePnPRansac` для большей устойчивости к шумным точкам;
- попробовать `SOLVEPNP_ITERATIVE` после хорошей инициализации;
- подогнать каноническую 3D-модель головы под реальные пропорции;
- отдельно нормировать модель под взрослого/ребёнка или под нужный домен.

### Работа с keypoints

- отбрасывать головы с плохим средним confidence;
- маскировать отдельные ключевые точки с низкой уверенностью;
- использовать fallback-логику, если одно ухо пропало;
- подключить отдельный face-landmarks детектор вместо pose-keypoints, если нужна более высокая точность.

### Realtime-стабильность

- добавить EMA/Kalman filter для `rvec`, `tvec` или уже для 2D-проекций;
- вести трекинг конкретного человека между кадрами;
- стабилизировать бокс под головной убор через temporal smoothing.

### Практическое применение

- проверять пересечение 3D-бокса с детекцией каски;
- использовать бокс как ROI для второго классификатора;
- строить правила вида: `helmet expected here`, `helmet missing`, `head turned away`.

## Ключевые функции в `face2head_live.py`

### `get_camera_matrix(w, h)`

Создаёт приближённую матрицу камеры по размеру кадра.

### `project_points(points_3d, rvec, tvec, K, dist)`

Проецирует 3D-точки в изображение через `cv2.projectPoints`.

### `head_pose_from_5pts(pts_2d_xy, K, dist)`

Центральная функция проекта.
Она:

- решает `PnP`;
- получает `R`, `rvec`, `tvec`;
- переводит основные и вспомогательные точки в систему камеры;
- проецирует `PTS_3D_AUX` и `PTS_3D_HAT` в 2D;
- вычисляет `face_vec` и `face_angle_deg`;
- возвращает всё это в словаре.

### `pick_best_person(result)`

Выбирает одного человека из результата pose-модели по среднему качеству head keypoints.

## Типичный сценарий использования

Практически это можно использовать так:

1. `YOLO pose` даёт 5 ключевых точек головы;
2. из них оценивается 6DoF-поза условной головы;
3. вокруг головы строится 3D-зона интереса;
4. дальше уже можно запускать проверку PPE, логику безопасности или аналитические правила.

То есть эти скрипты — не просто визуализация, а удобный геометрический мост между **2D keypoints** и **структурированным 3D ROI вокруг головы**.

## Замечания по устройствам

В коде по умолчанию указан:

```python
preferred_device = "mps"
```

Это логично для macOS/Apple Silicon.
Если `MPS` недоступен или падает, код автоматически переключается на `CPU`.

Для Linux/Windows с NVIDIA обычно имеет смысл заменить это на `cuda`, например вручную адаптировать блок выбора устройства.

## Диагностика проблем

### `Не удалось открыть вебкамеру`

Проверь:

- что камера не занята другим приложением;
- что `cv2.VideoCapture(0)` использует правильный индекс устройства;
- что у Python есть доступ к камере.

### `MPS ошибка ... переключаюсь на CPU`

Это штатное поведение fallback-механизма.
Если MPS нестабилен на конкретной конфигурации, можно сразу выставить:

```python
preferred_device = "cpu"
```

### `solvePnP failed`

Обычно это значит одно из следующего:

- keypoints слишком шумные или вырождены;
- точки перепутаны по порядку;
- часть головы плохо видна;
- каноническая 3D-модель слишком плохо соответствует наблюдаемой геометрии.

### Бокс рисуется странно

Почти всегда проблема в одном из трёх мест:

- неверный порядок 2D/3D-точек;
- слишком грубые интринсики;
- неподходящий масштаб `PTS_3D_HAT` относительно `PTS_3D`.

## Итог

`face2head.py` — это удобный sandbox для проверки геометрии на одном примере.
`face2head_live.py` — realtime-демо, которое показывает, как из 5 pose-keypoints получить оценку головы, достроить полезные 3D-точки и отрисовать область под головной убор поверх live-видео.

Для прототипирования задач PPE, helmet ROI и грубой head-pose аналитики это уже достаточно практичная и расширяемая база.
